🔴 Origen externo: Las OI no las crea el analista — las genera un sistema externo de monitoreo que las expone vía API/JSON. La herramienta levanta la OI automáticamente con estado Nueva. El analista solo puede resolverlas: Justificada o OS — Operación Sospechosa. El ROS se emite fuera de la herramienta.
⚠️ Las tablas en gris (sociedades_tag, usuarios, alertas) ya existen en el DER principal. Se muestran aquí solo para mostrar las relaciones con el módulo OI. La tabla alertas se actualiza con el campo id_oi.

Tabla: OPERACIONES_INUSUALES

🔴 Los campos marcados EXT vienen del sistema externo de monitoreo. No son editables por el analista.
CampoTipoDescripción
uuid PKUUIDIdentificador único interno de la OI
id_externoVARCHAR(50)ID asignado por el sistema externo de monitoreo. Ej: OI-2026-042 IDX
id_sociedad FKINTEGERSociedad/cliente a la que pertenece la operación → sociedades_tag.id_sociedad
id_analista_asignado FKINTEGERAnalista de Compliance asignado al caso → usuarios.id. Puede quedar NULL hasta asignación
tipo_operacion ENUMVARCHAR(30)Tipo de operación detectada: transferencia_bancaria / transferencia_virtual / efectivo / tarjeta / otro — viene del EXT
tipo_movimiento ENUMVARCHAR(20)ingreso / egreso — viene del EXT
montoDECIMAL(15,2)Monto de la operación detectada — viene del EXT
moneda ENUMVARCHAR(10)ARS / USD / digital — viene del EXT
fecha_operacionDATEFecha en que ocurrió la operación — viene del EXT
descripcion_externaTEXTDescripción/alerta generada por el sistema externo. Ej: "Monto 8x mayor al perfil transaccional"
estado ENUMVARCHAR(20)nueva / justificada / os (Operación Sospechosa). Una vez en justificada u os no puede modificarse
cuenta_bloqueadaBOOLEANSi la cuenta del cliente está bloqueada preventivamente por esta OI
fecha_bloqueoTIMESTAMPCuándo se bloqueó la cuenta. NULL si no fue bloqueada
fecha_desbloqueoTIMESTAMPCuándo se desbloqueó. NULL si sigue bloqueada
id_usuario_desbloqueo FKINTEGERAnalista que ejecutó el desbloqueo → usuarios.id
comentario_cierreTEXTComentario obligatorio al cerrar la OI (justificación o motivo OS). No puede estar vacío si estado ≠ nueva
created_atTIMESTAMPCuándo fue levantada por la herramienta (≠ fecha_operacion)
updated_atTIMESTAMPÚltima modificación
closed_atTIMESTAMPCuándo fue cerrada (justificada u OS). NULL si sigue abierta
Reglas de negocio clave
  • El analista no puede crear una OI manualmente — solo se insertan via integración con el sistema externo
  • Transición de estados: nueva → justificada | nueva → os. No hay vuelta atrás
  • comentario_cierre obligatorio: la BD debe tener un check constraint — NOT NULL cuando estado ≠ nueva
  • id_externo único: unique constraint en la BD para evitar duplicados de la integración

Flujo de Estados y Trazabilidad

Flujo de una OI — paso a paso

1
Sistema externo detecta operación → expone via API
2
Herramienta levanta la OI: estado = nueva · Primer registro en oi_log_estados
3
Se genera alerta oi_nueva en alertas
4
Analista revisa, adjunta evidencia (oi_adjuntos) y deja comentarios (oi_comentarios)
5
Analista decide cierre con comentario_cierre obligatorio
6
Estado cambia a justificada o os · Registro en oi_log_estados
7
Si es OS: ROS se emite fuera de la herramienta. OI queda cerrada con trazabilidad completa
