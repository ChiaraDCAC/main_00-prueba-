Contexto

La empresa se encuentra en un proceso de expansión de su modelo de negocio, con el objetivo de incorporar nuevas líneas de productos financieros, tales como lending y factoring.

Para poder operar bajo estos esquemas, resulta necesario constituirse como Proveedor de Servicios de Pago (PSP), lo cual implica cumplir con una serie de requerimientos regulatorios establecidos por organismos como el BCRA y la UIF.

En este contexto, surge la necesidad de conformar un equipo de Compliance que garantice el cumplimiento normativo en relación con:

  • Proceso de onboarding de usuarios
  • Validación y gestión de documentación
  • Monitoreo de operaciones
  • Generación de reportes regulatorios e informativos

Actualmente, estos procesos no cuentan con una herramienta centralizada, lo que genera falta de trazabilidad, dependencia de procesos manuales, riesgo de inconsistencias y dificultad para cumplir con auditorías.

Definición del Problema

Los PSP deben cumplir con requerimientos regulatorios estrictos de BCRA y UIF, que exigen recolección, verificación y conservación de información KYC, así como la gestión de procesos vinculados a PLA/FT.

Sin un sistema centralizado, los riesgos son:

  • Falta de centralización en la gestión y almacenamiento de documentación
  • Dificultades para auditar procesos de revisión, aprobación o rechazo
  • Riesgo de inconsistencias o errores en la información registrada
  • Mayor complejidad para cumplir con requerimientos regulatorios y responder auditorías

Casos de Uso

CU-00 — Alta de usuario y lectura de documentación

El analista de Compliance revisa y aprueba/rechaza la documentación específica según el tipo de sociedad. En caso de rechazo debe ingresar el motivo obligatoriamente. Una vez aprobada la documentación, carga los datos del cliente en el sistema persona por persona. Asigna riesgo y finaliza con la lectura del resumen de alta (modificable con comentarios). Si queda documentación o campos pendientes, el proceso queda en estado Pendiente de Alta.

Flujo detallado del CU-00:

1
Revisión de documentación: El analista recibe los documentos subidos por Atención al Cliente. Los revisa uno a uno y los aprueba o rechaza (con motivo obligatorio). Si toda la documentación obligatoria está aprobada, puede avanzar; si no, el proceso queda pendiente.
2
Carga de datos del legajo: Con la documentación aprobada como respaldo, el analista carga los campos de cada documento (estatuto, actas, registros) y los datos de cada persona vinculada a la sociedad.
3
Datos de personas — flujo detallado: Por cada persona se carga: datos identificatorios (nombre, apellido, DNI, CUIT, domicilio, etc.), rol en la sociedad (Presidente / Gerente / Director / Apoderado / Accionista / Beneficiario Final / Heredero / Socio SH), esquema de firma (Individual o Conjunta con alcance y participantes), PEP (Sí/No — Persona Expuesta Políticamente), y REPET (figura en el registro público). El sistema guarda automáticamente cada campo a medida que se completa.
4
Asignación de riesgo: El analista evalúa el perfil del cliente y asigna nivel de riesgo (Bajo / Medio / Alto) con el tipo de debida diligencia correspondiente (simplificada / media / reforzada). Este proceso es manual con asistencia opcional del sistema.
5
Resumen y alta: El analista lee el resumen completo del cliente, puede dejar comentarios y da de alta al usuario. Si hay documentación o campos faltantes, el estado queda en Pendiente de Alta con alertas de seguimiento.
6
POST a BD DCAC: Una vez dado de alta, el sistema postea a la base de datos interna: estado de la cuenta (activa/pendiente/rechazada), datos de personas, roles, esquemas de firma, nivel de riesgo y tipo de debida diligencia.
CU-01 — Envío de reportes regulatorios

El analista genera los reportes exigidos por BCRA/UIF (Base Padrón, Apartados A/B/C, ARCA F.8126), consolidando información del sistema. Incluye validaciones automáticas, revisión manual y envío dentro de plazos regulatorios. El sistema permite versionado, almacenamiento histórico, trazabilidad y carga de comprobantes de envío.

CU-02 — Definición de riesgo

El analista evalúa el perfil del cliente en base a múltiples variables y asigna un nivel de riesgo (Bajo/Medio/Alto). En función del análisis, determina el tipo de debida diligencia (simplificada/media/reforzada). La decisión final queda registrada con justificación, asegurando trazabilidad y cumplimiento normativo. El sistema puede asistir mediante reglas o scoring automático.

CU-03 — Clasificación de operaciones inusuales

El analista monitorea transacciones potencialmente inusuales. El sistema genera un caso y asocia la transacción al legajo del cliente. La cuenta puede bloquearse preventivamente. Atención al Cliente solicita y carga la documentación respaldatoria. El analista analiza, registra su análisis y decisión. Si la operación es justificada, se puede levantar el bloqueo. Si no, se escala como Operación Sospechosa. El ROS se gestiona por fuera de la herramienta conforme normativa.

CU-04 — Gestión de clientes

El analista accede al legajo completo del cliente: información, documentación, nivel de riesgo, estado de cuenta y alertas pendientes. Permite seguimiento integral, identificación de vencimientos, revisión de observaciones y control del estado de habilitación operativa.

CU-05 — Gestión de alertas

El analista monitorea y gestiona alertas vinculadas con vencimientos de documentación, altas pendientes y observaciones. El sistema genera alertas automáticas por eventos definidos. Las alertas pueden escalarse y ser visualizadas por otros roles (Atención al Cliente), asegurando gestión coordinada.

CU-06 — Alta con documentación incompleta (excepción) NUEVO

El analista puede dar de alta una cuenta aunque no cuente con toda la documentación obligatoria, marcándola como Alta con Excepción. El sistema registra qué documentación falta, establece un plazo máximo para completarla y genera alertas automáticas de seguimiento. El analista debe dejar justificación del motivo de la excepción. Estas cuentas quedan visibles en una vista separada de "Excepciones pendientes".

CU-07 — POST de acciones clave a BD DCAC NUEVO

El sistema registra en la BD interna todas las acciones críticas de compliance para mantener trazabilidad y permitir auditorías:

  • Alta de cuenta: postea estado, fecha, analista responsable, nivel de riesgo, tipo de debida diligencia
  • Rechazo de cuenta: postea motivo, analista, fecha
  • Bloqueo/desbloqueo: postea tipo de acción, motivo, analista, fecha
  • Cambio de riesgo: actualiza el campo de riesgo dentro del registro existente del cliente (no es un nuevo registro)
  • Aprobación/rechazo de documento: postea estado, motivo de rechazo si aplica, analista, fecha

Todas estas acciones quedan en el log de acciones vinculado al legajo del cliente.

RF-35 Observaciones del cliente ✓ El legajo permite registrar observaciones libres sobre el cliente.

RF-41 NUEVO Alta con excepción ✓ El analista puede marcar el alta como "Excepción" indicando qué documentos faltan y un plazo para completarlos.
✓ El sistema genera alertas automáticas a medida que se acerca el plazo.
✓ Las cuentas en excepción aparecen en una vista separada claramente identificada.
✓ El motivo de la excepción queda registrado y es auditabe.
RF-42 NUEVO POST a BD DCAC ✓ Alta de cuenta: se postea estado, fecha, analista, riesgo y tipo de debida diligencia al registro del cliente en BD DCAC.
✓ Rechazo: se postea motivo, analista y fecha.
✓ Bloqueo/desbloqueo: se actualiza el estado de la cuenta en el registro existente (no se crea un registro nuevo).
✓ Cambio de riesgo: se actualiza el campo riesgo en el registro existente.
✓ Cada acción genera una entrada en el log de acciones vinculado al cliente.
